### Added

- **Concept Ideation front door** — new mandatory `concept-ideator` agent (`skills/dify/agents/concept-ideator.md`) + SKILL.md **Step 2b** confirmation gate. Every forward-direction request now starts by aggressively expanding the user's prompt into a rich **App Concept Proposal** (vision, a generous set of typed inputs, multiple output types, optional creative features), which the user confirms or edits before the pipeline runs. Ripple edits: `requirements-analyzer.md` now treats the confirmed concept as authoritative scope; `SKILL.md` Hard Rules updated (concept-ideator added to mandatory agents; new gate rule 11); `CLAUDE.md` pipeline renumbered + agent count 13 -> 14. - **Knowledge Base ingestion pattern** — new capability: building workflows that **write/upsert into** a Dify KB via the Dataset API (the inverse of RAG retrieval).
  - **`references/patterns/knowledge-base-ingestion.md`** (new) — mode-agnostic pattern doc: how to choose the chunking/index mode (general `text_model` / parent-child `hierarchical_model` / Q&A `qa_model`) from the data; **how to choose the parent and child delimiters** by inspecting the extracted text (parent = what's returned as context, child = what the query matches); the Dataset write-API surface (`create-by-text`, `create-by-file`, list/create/update segments, pagination); the idempotent upsert topology (validate → parse → check existence → branch create/update → diff → iterate); env-var + SETUP.md guidance.   - **`assets/workflows/knowledge-base-ingestion.yml`** (new) — sanitized real Dify export (from the user's `DX_Suite_KB` workflow) used as the ground-truth **syntax reference** for the KB write API calls/payloads; secrets and instance domain scrubbed, header comment marks it a syntax reference (not the canonical topology).

### Changed

- **Code-node `outputs` switched to the dict/mapping format** (G2) — real Dify exports represent code-node outputs as a dict keyed by variable name (`var_name: {type, children: null}`), not the list form the skill assumed. Verified against two unrelated ground-truth exports (12/12 code nodes use the dict form; `children` is always `null`, even for `array[object]`). Changes:
  - `validate_workflow.py` — code-node check now **requires the dict form** (each entry must have a valid `type` from the 7 allowed types + a `children` key) and **rejects the old list form**; the separate end-node outputs check (which legitimately uses a list) is untouched. Also fixed a Windows **cp932 `UnicodeEncodeError`** crash by forcing UTF-8 stdout, and broadened the entry-node check to accept `trigger-schedule` / `trigger-webhook` (not only `start`) so ground-truth trigger workflows validate.
  - `references/nodes/code.md` — rewrote the outputs section to the dict form, documented all 7 output types and `children: null`, and added an **"always return every declared output on every code path"** rule (init defaults up front, return empties on error) with a before/after example.
  - `agents/dsl-generator.md` — code-node example now emits the dict form + the always-return guidance.
  - `scripts/fix_assets.py` — reversed its outputs migration (was converting the *correct* dict form back to list); now converts old list → dict with `children: null`.
  - Assets/tests — converted `assets/workflows/webhook-triggered.yml` to the dict form; added pytest cases (list rejected, invalid type caught, `array[object]` accepted). **All 13 assets validate; full suite 97/97 green.**
