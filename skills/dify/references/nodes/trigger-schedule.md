# Trigger: Schedule Node

## Overview

The Schedule Trigger node (`trigger-schedule`) starts a workflow automatically on a time-based schedule. It replaces the Start node as the workflow entry point. No user action or API call is needed — Dify fires the workflow on the configured cron schedule.

Use it for: daily reports, recurring data cleanup, regular health checks, scheduled notifications, and any time-driven automation.

Each workflow can have **at most one** Schedule Trigger node. It is available only in **Workflow** apps (not Chatflow).

## Node Type

```
type: trigger-schedule
```

In edge data, `sourceType: trigger-schedule`.

## Configuration Fields

| Field | Type | Description |
|-------|------|-------------|
| `cron_expression` | string | Standard 5-field cron: `Minute Hour Day Month Weekday` |
| `mode` | string | `cron` — use the cron expression as the schedule |
| `timezone` | string | IANA timezone string (e.g. `UTC`, `Asia/Tokyo`, `America/New_York`) |
| `config` | object | Visual UI representation of the schedule (see below) |

### `config` block

The `config` block mirrors what the visual schedule editor shows. It is generated by Dify and does not affect execution — `cron_expression` is what actually runs.

```yaml
config:
  frequency: daily        # daily | weekly | monthly | hourly
  mode: visual
  timezone: UTC
  visual_config:
    monthly_days:
    - 1                   # day of month (1-31)
    on_minute: 0          # minute offset
    time: 12:00 AM        # human-readable time
    weekdays:
    - sun                 # sun | mon | tue | wed | thu | fri | sat
```

## Cron Expression Format

Standard 5-field Unix cron — `Minute Hour Day-of-Month Month Day-of-Week`:

| Expression | Meaning |
|------------|---------|
| `0 9 * * *` | Every day at 9:00 AM |
| `0 9 * * 1` | Every Monday at 9:00 AM |
| `0 */6 * * *` | Every 6 hours |
| `30 8 1 * *` | 1st of every month at 8:30 AM |
| `0 9 * * 1-5` | Weekdays at 9:00 AM |

## Exposed Variables

The Schedule Trigger produces no custom output variables. Only the system variable is available downstream:

| Variable | Type | Description |
|----------|------|-------------|
| `{{#sys.timestamp#}}` | string | ISO timestamp of when the workflow was triggered |

## Complete YAML Example

```yaml
- data:
    config:
      frequency: daily
      mode: visual
      timezone: UTC
      visual_config:
        monthly_days:
        - 1
        on_minute: 0
        time: 12:00 AM
        weekdays:
        - sun
    cron_expression: 0 9 * * *
    mode: cron
    selected: false
    timezone: UTC
    title: Schedule Trigger
    type: trigger-schedule
  height: 130
  id: '[node_id]'
  position:
    x: 80
    y: 282
  positionAbsolute:
    x: 80
    y: 282
  selected: false
  sourcePosition: right
  targetPosition: left
  type: custom
  width: 242
```

## Common Mistakes

1. **Using Start node instead.** Workflows meant to run on a schedule must use `type: trigger-schedule`, not `type: start`. A Start node requires manual execution.

2. **Wrong timezone field.** The top-level `timezone` field controls when the cron fires. The `config.timezone` is only for the visual UI and does not affect execution.

3. **Adding a second Schedule Trigger.** Only one `trigger-schedule` node is allowed per workflow. Dify will reject a second one.

4. **Expecting output variables.** The Schedule Trigger produces no custom variables — only `{{#sys.timestamp#}}` is available downstream.
