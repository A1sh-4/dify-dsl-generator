# Dify Instance Setup

This guide covers how to get a Dify instance running — either using Dify Cloud or a self-hosted Docker deployment — and how to prepare it for use with the Dify DSL Generator plugin.

---

## Dify Cloud vs Self-Hosted

Choose the option that fits your needs:

| | Dify Cloud | Self-Hosted |
|---|---|---|
| Setup time | 5 minutes | 30–60 minutes |
| Maintenance | Managed by Dify | Your responsibility |
| Cost | Free tier + paid plans | Infrastructure cost only |
| Data privacy | Dify's servers | Your own servers |
| Customization | Limited to UI settings | Full — config files, plugins, storage |
| Scaling | Automatic | Manual |

Both options are fully compatible with DSL files generated by this plugin.

---

## Option A: Dify Cloud

1. Go to [dify.ai](https://dify.ai) and click **Get Started**.
2. Sign up with email, Google, or GitHub.
3. Create a **workspace** (or accept an invitation to join an existing one).
4. Your workspace URL is `https://app.dify.ai` — all apps and knowledge bases live here.
5. Get your workspace API key: **Settings → API Keys → Create API Key**. Store this key; you will need it if you use the datasets API to manage knowledge bases programmatically.

Dify Cloud is the fastest way to get started. The free tier supports a limited number of workflow runs per month and a limited storage quota for knowledge bases. Upgrade to a paid plan when you need higher limits.

## Option B: Self-Hosted (Docker Compose)

### Prerequisites

- **2 CPU cores, 4 GB RAM** minimum (4 cores, 8 GB recommended for production)
- **20 GB storage** minimum
- **Docker** and **Docker Compose** installed
- A machine with internet access (for pulling images and model API calls)

### Installation Steps

```bash
# Clone the Dify repository
git clone https://github.com/langgenius/dify
cd dify/docker

# Copy the example environment file
cp .env.example .env
```

Open `.env` in a text editor and set the following at minimum:

```env
# Required: generate with: openssl rand -hex 32
SECRET_KEY=your-random-secret-key-here

# Database credentials (PostgreSQL)
DB_USERNAME=dify
DB_PASSWORD=your-db-password
DB_DATABASE=dify

# Redis connection
REDIS_URL=redis://:your-redis-password@redis:6379/0
REDIS_PASSWORD=your-redis-password
```

Then start all services:

```bash
docker compose up -d
```

Dify will be available at `http://localhost` (port 80 by default). The first startup takes a few minutes while containers initialize and database migrations run.

Open your browser, go to `http://localhost`, and complete the admin account setup wizard.

### Key `.env` Configuration Reference

| Variable | Purpose | Example value |
|---|---|---|
| `SECRET_KEY` | Flask session signing key | `openssl rand -hex 32` output |
| `DB_USERNAME` | PostgreSQL username | `dify` |
| `DB_PASSWORD` | PostgreSQL password | strong random string |
| `REDIS_URL` | Redis connection string | `redis://:pass@redis:6379/0` |
| `STORAGE_TYPE` | File storage backend | `local` or `s3` |
| `FILES_URL` | Public URL for serving files | `http://localhost` |
| `CONSOLE_URL` | URL for the web console | `http://localhost` |

For production deployments, also configure:
- `STORAGE_TYPE=s3` with your S3 bucket credentials
- A proper domain and TLS termination (nginx or a load balancer in front of Docker)
- Outbound email settings (`MAIL_TYPE`, `SMTP_*`) for account verification

### Updating a Self-Hosted Instance

```bash
git pull
docker compose pull
docker compose up -d
```

## Required First Steps After Dify is Running

Regardless of Cloud or self-hosted, complete these steps before importing any DSL files:

### 1. Add a Model Provider

Dify requires at least one LLM provider to execute any workflow or chatflow. Without this, all workflow runs will fail immediately.

Go to **Settings → Model Provider → Add provider**. Add at minimum one text generation model (e.g., Anthropic Claude or OpenAI GPT-4o). See `docs/setup/model-configuration.md` for detailed provider setup instructions.

### 2. Create a Knowledge Base (RAG workflows only)

If your workflow includes a knowledge retrieval node, you need at least one knowledge base with indexed documents before the workflow can run.

Go to **Knowledge → Create Knowledge Base**. Upload at least one document and wait for indexing to complete (status changes from "Indexing" to "Available"). See `docs/setup/knowledge-base-setup.md` for full details.

### 3. Install Marketplace Plugins (if your workflow uses plugin nodes)

If the DSL file references any Dify marketplace plugins (e.g., Brave Search, Firecrawl, Jina), those plugins must be installed in your workspace before the workflow can run.

Go to **Plugin → Marketplace**, search for the plugin by name, and click **Install**.

---

## Getting the Workspace API Key

For knowledge base operations via the API (creating datasets, uploading documents programmatically), you need the workspace-level API key — not an app-level API key.

**Dify Cloud:** Settings → API Keys → Copy the key labeled "Workspace API Key"

**Self-hosted:** Settings → API Keys → Create a new key (workspace scope)

This key is used in the `Authorization: Bearer <key>` header for requests to the `/v1/datasets` API endpoints.

---

## Verifying the Instance is Ready

Before using the plugin to generate and import DSL files, confirm:

- You can log in to the Dify console
- At least one model provider shows as "Active" in Settings → Model Provider
- The "Studio" or "Apps" section is accessible
- You can click "Create App" without errors

If all of these work, your Dify instance is ready to receive imported DSL files.
